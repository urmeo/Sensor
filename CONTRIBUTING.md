# Contributing to Sensor

Thanks for your interest. Sensor is a small, single-author resource: a review of
the sensors and software platforms used in our anxiety-detection research, plus a
sample dataset and one analysis notebook. Contributions that improve accuracy,
fix data or documentation errors, or extend the notebook are welcome.

## Development setup

Sensor targets Python 3.11+ (CI tests 3.11 and 3.12). For a reproducible,
hash-locked environment, install from the lockfile — it pins every dependency,
including the dev and notebook tooling:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.lock
pip install -e . --no-deps          # the sensor_data package + scripts
```

`requirements.lock` is generated with `uv pip compile pyproject.toml --all-extras
--generate-hashes` and is the source of truth for CI. If you only need the runtime
deps, `pip install -e ".[dev]"` also works.

## Running the checks

Run these from the repository root before opening a pull request. CI runs the
same commands.

```bash
pytest tests/ -q                     # data contracts, integrity, derive/shift logic
ruff check .                         # lint
ruff format --check .                # formatting
mypy sensor_data.py scripts          # type check
```

Because this is a notebook-based project, CI also executes the notebook
end-to-end and fails if any cell errors. Run the same smoke test locally after
touching `analysis/explore_data.ipynb`:

```bash
pytest --nbmake --nbmake-timeout=600 analysis/explore_data.ipynb
```

Commit the notebook only after a clean top-to-bottom run: `tests/test_notebooks.py`
rejects any notebook committed with saved error outputs.

## Changing the data

The CSVs in `data/` are covered by `tests/test_data_contract.py`. If you edit a
file or its schema, update the test and the [data dictionary](data/DATA_DICTIONARY.md)
in the same change so the two stay in sync. The de-identified `data/relative/`
variant is generated by `scripts/shift_time.py` and is gitignored — don't commit it.

## Style

- Clean code, few comments: let names and structure carry the meaning.
- Keep functions short and single-purpose; ruff is configured for a 100-char line.
- Notebook cells stay tidy — one idea per cell, no dead scratch code.

## Commit messages

This repo uses [Conventional Commits](https://www.conventionalcommits.org): a
`type:` prefix and a terse, lowercase description. Use `feat:`, `fix:`, `docs:`,
`test:`, `refactor:`, `build:`, or `chore:` (for example `fix: correct RMSSD
threshold source` or `docs: clarify data ethics`). Keep one logical change per commit.

## Filing an issue

Open an issue for anything wrong or missing — a mislabelled sensor spec, a broken
threshold reference, a data-dictionary mismatch, or a notebook that won't run.
Include the file and, for data or notebook problems, enough to reproduce it.

## Opening a pull request

1. Branch off `main`.
2. Make your change and keep commits focused.
3. Run the checks above (tests, ruff, mypy, and the notebook smoke test) locally.
4. Open a pull request against `main`. CI must be green before merge.
